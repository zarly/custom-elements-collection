# Core / generator / schema maintenance

Rare, high-blast-radius work. **Every component downstream is affected.** Bisect-safety is paramount: `pnpm check` is the floor, but it doesn't notice cross-file drift between meta JSON and generator outputs ‚Äî you have to think about both sides of every change.

> If the change is "add a feature to one component", that's not core maintenance ‚Äî load `cec-component-author`. Core maintenance is about the contracts those components inherit: the base class, the schema, the generators, the build, the ADR record.

---

## When to use

- Editing `src/core/` (`CecElement` base, `json-prop.ts`, `define-once.ts`, helpers).
- Editing `src/meta/schema.ts`, `src/meta/groups.ts`, `src/meta/tiers.ts`, `src/meta/types.ts`, `src/meta/registry-types.ts`.
- Editing or adding a script in `scripts/`: `generate-exports.ts`, `generate-registry.ts`, `generate-skill.ts`, `validate-meta.ts`, `sync-meta-dates.ts`, `build-publish-manifest.ts`, `bundle-stats.ts`, `pre-commit-hook.sh`.
- Editing `vite.config.ts`, `vitest.config.ts`, `playwright.config.ts`, `tsconfig.json`, `tsconfig.build.json`, `eslint.config.mjs`, `stylelint.config.mjs`.
- Editing `src/tokens/tokens.css` (the base layer ‚Äî adds or removes a token across the whole catalog).
- Writing a new ADR or amending an accepted one.
- Promoting a CDR to a validator rule (the lifecycle final step).

## When NOT to use

- Adding a `--ce-color-*` or `--ce-radius-*` override in one theme bundle ‚Üí `cec-theming`. The base file `tokens.css` is core; the per-school bundles `src/tokens/<slug>.css` are theming.
- Per-component prop / event / slot changes ‚Üí `cec-component-author`.
- Release sequencing ‚Üí `cec-publishing`.

## The generated / derived inventory ‚Äî never hand-edit

A change to a meta JSON or to a generator should leave every artefact below regenerated by `pnpm gen-*`. Editing any of these by hand is rejected at the `gen-skill:check` gate in `pnpm check`.

| Generated artefact | Source of truth | Generator |
|---|---|---|
| `src/index.ts` | every `*.meta.json` | `generate-exports.ts` |
| `src/auto.ts` | every `*.meta.json` | `generate-exports.ts` |
| `src/entries/<tag>.ts` | each component's meta | `generate-exports.ts` |
| `src/manifest.ts` | every `*.meta.json` | `generate-exports.ts` |
| `src/manifest.publish.ts` | non-`internal` metas | `build-publish-manifest.ts` |
| `package.json#exports` | every published subpath | `generate-exports.ts` |
| `../cec-consumer/references/catalog.md` (whole file) | every `*.meta.json` | `generate-skill.ts` |
| `../cec-consumer/references/index.md` (whole file) | every `*.meta.json` | `generate-skill.ts` |
| `dist/registry.json` + `dist/registry/<tag>.json` + filtered views | every meta + tier filter | `generate-registry.ts` (Vite plugin `copyRegistry()`) |
| `dist/meta/<tag>.json` + `dist/meta/index.json` | meta files | Vite plugin in `vite.config.ts` |
| `src/meta/content-hashes.json` | SHA-256 of each `<stem>.ts` | `sync-meta-dates.ts` via pre-commit hook |
| `meta.created` / `meta.updated` | content-hash registry + git timestamp | `sync-meta-dates.ts` (ADR-011/012) |

If a generator output looks wrong, the bug is in the generator or the meta ‚Äî never patch the output.

## Changes to `src/meta/schema.ts` (Zod) ‚Äî the highest-impact lever

Adding a field, tightening a type, or adding a Zod refinement reshapes every component's meta on the next `pnpm validate-meta`. Procedure:

1. **Add the field as optional first.** Mass-rejecting every meta with a single PR is bad bisect-safety. Optional ‚Üí backfill in a separate pass ‚Üí tighten to required only after all metas pass the new shape.
2. **Update `src/meta/types.ts`** in the same change so consumers importing types stay accurate.
3. **Run `pnpm validate-meta`** ‚Äî should show 0 failures on optional, expected failures on tightening.
4. **If the new field surfaces in the LLM-tool-use registry, update `src/meta/registry-types.ts` + `scripts/generate-registry.ts`** to project it.
5. **Update CONTRIBUTING.md ¬ß4** and [`docs/protocols/new-component.md`](../../docs/protocols/new-component.md) step 4 if the field is contributor-facing.
6. **Write an ADR** if the change is structural (a new field kind, a new closed-enum, a new validation invariant). The ADR numbers are sequential ‚Äî last accepted is **ADR-014** (Streaming-lifecycle conformance lint surface). The next is 015.

The same procedure applies to adding a value to a closed enum (e.g. `stability`, `tier`, `category`, `sideEffects.kind`, `cssVariables.kind`) ‚Äî additive first, ADR if structural.

## Changes to `src/meta/groups.ts` (catalog grouping)

The first entry of `meta.tags[]` is constrained to this allow-list. Adding a group:

1. Append the new group to `src/meta/groups.ts`.
2. Update [`../../docs/protocols/pre-release.md`](../../docs/protocols/pre-release.md) step 6 ‚Äî the `e2e/catalog-smoke.spec.ts` `REPRESENTATIVE_TAGS` list adds an entry per group.
3. Update [`../../README.md`](../../README.md) "Catalog" section if it groups by manifest group.
4. Run `pnpm gen-skill` ‚Äî the auto-generated `../cec-consumer/references/catalog.md` rebuilds with the new group.

Removing or renaming a group is a **major** change ‚Äî it shifts existing metas. Treat as ADR-bound.

## Adding a new tag prefix or build mode ‚Äî ADR required

The only accepted prefixes are `ce-*` and `lesson-*`. A new prefix touches generator regex, ESLint rules, Stylelint scope, manifest grouping, skill catalog, and bundler chunk naming. **Open an ADR before writing the code.** Same gate for: adding a new distribution mode (today: inline / linked-local / linked-CDN per ADR-004), changing the `prepublishOnly` chain, or adding a peer dependency.

ADRs and their numbering live in [`../../docs/adr/`](../../docs/adr/). Template lives in the existing ADRs ‚Äî none of them have an explicit template file; copy the most-recent (ADR-014) and adapt.

## Tokens ‚Äî base layer vs theme bundles

`src/tokens/tokens.css` is **core**: adding or removing a `--ce-*` token there affects every component and every theme. `src/tokens/<slug>.css` is **theming**: that's `cec-theming` territory.

When adding a new token to `tokens.css`:

1. Add the variable with a sensible dark-default value.
2. Add the matching declaration to **every** existing theme bundle that diverges from the default. Missing values silently fall back to dark ‚Äî usually wrong.
3. Update [`../cec-theming/references/tokens.md`](../cec-theming/references/tokens.md) ‚Äî it enumerates the canonical axes.
4. Decide whether the new token is a "geometry" axis (`--ce-space-*`, `--ce-inset-*`, `--ce-sz-*`, `--ce-text-*` size scale) or an "appearance" axis (colour, shadow, font face). Geometry axes are **not** themable ‚Äî themes change appearance, not geometry. Document the axis in the ADR-003 lineage if you're adding a new kind.
5. ADR if structural (a whole new axis, not just a colour); skip if it's a sibling of an existing axis (one more semantic colour, one more spacing step).

## Hooks, ESLint, Stylelint ‚Äî the silent gates

`scripts/pre-commit-hook.sh` runs `sync-meta-dates` on every commit. Bypassing it with `SKIP_SIMPLE_GIT_HOOKS=1` leaves `content-hashes.json` stale; `sync-meta-dates:check` fails next release. Don't bypass unless debugging ‚Äî and re-run `pnpm sync-meta-dates` before committing.

ESLint enforces ADR-013 (the safety contract: no `innerHTML`, no `eval`, no `unsafeHTML`, console policy) and ADR-014 (streaming lifecycle). Stylelint enforces ADR-003 (no hex / named colours in components) and ADR-001 (no `@import` at runtime in component CSS).

When lint surfaces an issue you haven't seen before, follow [`../../docs/protocols/lint-fix.md`](../../docs/protocols/lint-fix.md) ‚Äî the per-rule actions live in [`../../docs/decisions/eslint.md`](../../docs/decisions/eslint.md) and [`../../docs/decisions/stylelint.md`](../../docs/decisions/stylelint.md). Apply documented actions without asking; for un-decided rules, propose with named edge cases (multi-theme semantic mismatch, alpha channels, etc.) and write the operator's decision back to the table.

## Bundler / typescript / vitest config changes

- **`vite.config.ts`** owns the `copyTokens()`, `copyMeta()`, and `copyRegistry()` plugins. If you add a new "side-product" emitted into `dist/`, model it as a plugin here (don't add a step to `prepublishOnly`).
- **`tsconfig.build.json`** controls `.d.ts` emission; `tsconfig.json` controls the editor / `typecheck` baseline. Differences are deliberate; align them only when you understand why they differ.
- **`vitest.config.ts`** runs jsdom for component unit tests. Don't switch to `happy-dom` without measuring ‚Äî jsdom is the regression baseline; behaviour parity is not guaranteed.
- **`playwright.config.ts`** boots `pnpm demo` on port 4600 with a `webServer` block. If port 4600 / 4601 are claimed (multi-agent host), pass `--port 4612` and update the spec retry count if flake increases.

## ADR discipline

- **Numbering is monotonic.** Don't reuse retired numbers; mark superseded ADRs with a `Status: superseded by ADR-NNN` header rather than deleting them.
- **MUST level.** ADRs are bright lines ‚Äî violations need an explicit override ADR (which itself becomes ADR-N+1). Don't soften an ADR by adding "unless ‚Ä¶" ‚Äî write a new ADR that supersedes it.
- **Cross-link both ways.** A new ADR that touches an existing one updates the prior ADR's `Relates-to:` block (or its body, if there's existing prose) so the lineage is discoverable from either side.
- **Audit the meta schema.** Every ADR that adds a new invariant should be enforced ‚Äî `validate-meta`, lint, generator, or pre-release protocol step. An ADR with no enforcement is decorative.

The lifecycle `candidate (CDR) ‚Üí accepted (CDR) ‚Üí distilled to validator rule` for daily-design conventions lives in [`../../docs/cdr/README.md`](../../docs/cdr/README.md). The promotion step (CDR ‚Üí validator) is core work ‚Äî you're updating either the validator skill or the schema.

## Verify on every change

```bash
pnpm check          # the bisect-safe floor
pnpm test:e2e       # if you touched anything that affects the demo / hydration / upgrade order
pnpm pack:inspect   # if you changed the build, tarball shape, or sideEffects array
```

If `validate-meta` regresses across components after a schema change, the schema change is too aggressive ‚Äî soften (make optional, refine, then tighten across a series of commits) rather than mass-fixing metas in a single commit.

---

## Anti-patterns (specific to this layer)

- **Patching a generator output.** Drift surfaces on the next `pnpm check`. Fix the generator.
- **Mass meta edits to chase a tightened schema.** Either the schema is wrong or the change should land in two steps (optional ‚Üí required).
- **New peer dependency at runtime.** Lit is bundled; `dist/` is self-sufficient. Anything else is ADR-bound.
- **Editing the registry projection in place** instead of updating `scripts/generate-registry.ts`. The projection is deterministic ‚Äî regenerate, don't patch.
- **New ESLint / Stylelint rule without writing to the decisions file.** The protocol explicitly trades operator-time-paid-twice for one-time decision capture; skipping it costs more later.
- **Changing `package.json#files`** without updating the tarball-allow-list check in [`../../docs/protocols/pre-release.md`](../../docs/protocols/pre-release.md) step 5.
- **`SKIP_SIMPLE_GIT_HOOKS=1` for routine commits.** Use it only when actively debugging the hook itself, then re-run `pnpm sync-meta-dates` before committing.
